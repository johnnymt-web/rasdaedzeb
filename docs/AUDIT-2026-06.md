## 1. Executive Summary

Pathfinder is a React 18 + Vite + TypeScript SPA on Supabase (Postgres + Auth + 11 edge functions) deployed via Vercel, serving career guidance to students aged ~12–18 across six psychometric instruments with AI-generated synthesis.

**Overall posture: solid security foundation, with specific, addressable gaps.** Row-Level Security is comprehensively enabled, AI keys are server-side, AI prompts have strong child-safety guardrails, and assessment scoring for three instruments is now tamper-proof at the database. The material gaps are: (a) **privacy/consent** for minors' data sent to third-party AI, (b) **authenticated cost-abuse** exposure on AI utility functions, (c) **code↔schema drift** that has the typecheck gate red, and (d) **migration non-reproducibility**.

**Top priorities:**
1. 🟠 Add a **parental/data-processing consent** mechanism for AI processing of minors' data (§6).
2. 🟡 Add **rate-limiting / tighter checks** to open AI utility functions (§4).
3. 🟡 Restore a **green typecheck gate** and resolve the **`superadmin`** architecture gap (§3, §7).
4. 🟡 Make migrations **reproducible** (baseline/squash legacy SQL) (§2).
5. 🟡 Complete **Phase B** so RIASEC/Skills/EQ scoring is server-authoritative (§5).

## 3. Database & Schema

✅ **43 tables, RLS enabled on 43/43, zero tables without policies.** Strong baseline.
✅ **`app_role` enum = `{student, parent, counselor, admin}`.**

🟡 **F-SCHEMA-1 — `superadmin` is a frontend role with no DB backing.** ✅ The enum has no `superadmin`, yet the frontend wires it throughout: `/superadmin` routes gated `allowedRoles={["superadmin"]}` (`App.tsx`), `useAuth` role union, sidebars/headers, and a full `SuperAdminDashboard`. **In production the entire `/superadmin` area is unreachable** (no user can hold the role); it is reachable only via the dev-only override. This is a half-built privileged feature.
→ *Decision required:* either **(A)** make `superadmin` real (add enum value + provisioning + RLS review — an auth-privilege change), or **(B)** treat it as dev-only scaffolding and make the code honest.

🟢 **F-SCHEMA-2 — `schools` phantom columns.** ✅ Table columns are `{id, name, is_default, created_at}` only; address & contact_email are **encoded inside the `name` string** (`"Name (addr | email)"`, parsed at runtime). `SchoolManagement.tsx` referenced non-existent `s.address`/`s.contact_email`. Fragile design; fix staged on `fix/typecheck-debt`.

## 4. Edge Functions & Access Control

✅ **Privileged functions verify identity + role:** `bulk-onboard-users`, `admin-insights` (admin), `counselor-coach` (counselor), `parent-coach`, `generate-synthesis` (access guard + caller check + 40/day rate limit). All AI/provider keys read from `Deno.env` — **never exposed to the frontend**. ✅

🟡 **F-FUNC-1 — AI utility functions lack role/ownership checks + rate limiting.** ⓘ `career-coach`, `localize-careers`, and `generate-parent-insight` perform **no in-code auth check**. `config.toml` sets no per-function `verify_jwt`, so the **platform default (valid JWT required) applies** — these are *not* fully public, but any **authenticated** user can invoke them with caller-supplied input, with **no rate limiting**. Risk: **authenticated cost-abuse / AI-quota exhaustion** (financial + availability), and `generate-parent-insight` will summarize arbitrary caller-supplied data for any role.
→ *Remediation:* add per-user/day rate limiting (as `generate-synthesis` already does); for `generate-parent-insight`, verify the caller is the linked parent/owner and pull data server-side rather than trusting the body. *(Confirm `verify_jwt` status in the dashboard; if any are set to `false`, severity rises to 🟠.)*

⚪ **F-FUNC-2 — `refresh-onet-cache` / `onet-proxy`.** ⓘ Service-role cache refresh (cron-style) and an O*NET API proxy; low data-sensitivity but worth confirming they aren't openly invokable.

## 5. Assessment Integrity (the G5 program)

✅ **F-ASSESS-1 — Phase A is live & tamper-proof.** `g5_score_*` + `g5_apply_*` functions and `g5_rescore_*` BEFORE-triggers exist on `big_five_assessments`, `caas_assessments`, `work_values_assessments`; scores are recomputed server-side from raw responses (parity-validated, self-test confirmed). Client-sent scores for these three cannot be forged.

🟡 **F-ASSESS-2 — RIASEC / Skills / EQ are still client-trusted.** ⓘ✅ These write to `assessments` via direct client insert (RLS `is_self` allows it); the `results` JSON is whatever the client sends. **Phase B** (server-authoritative `submit-assessment` edge function + RLS lockdown) is built and staged on `g5-phase-b` (vitest parity green) but **not yet live**. Until Phase B ships, a modified client can store fabricated RIASEC/Skills/EQ results.
→ *Remediation:* complete the staged Phase B rollout (already gated and in progress).

## 6. AI & Data Handling for Minors

⚪ **F-AI-1 (strength) — Excellent age-appropriate guardrails.** AI system prompts explicitly forbid diagnostic/prescriptive language, require "your responses suggest…", and steer students (ages 12–18) toward counselors/parents. This is good child-safety practice and should be preserved.

🟠 **F-AI-2 — No consent / data-processing mechanism for minors' data sent to third-party AI.** ⓘ Assessment results, student names, and report context are sent to **OpenAI** (gpt-4o-mini: coaches, parent-insight, localize), **Anthropic** (synthesis), and a **Lovable AI gateway** (admin/counselor/parent coaches). There is **no consent flow, no privacy gate, and no documented DPA** for processing children's personal/psychometric data through these processors. Depending on jurisdiction (EU GDPR, and Georgia's data-protection law for a Georgian-market product), this is a material compliance exposure.
→ *Remediation:* add explicit parental/guardian consent capture at onboarding; document processors + DPAs; consider data-minimization (send the minimum needed, avoid names where possible); add a privacy policy surface.

🟢 **F-AI-3 — Provider sprawl + uneven rate limiting.** Three AI providers; only `generate-synthesis` is rate-limited/cached. Consolidate where feasible and apply consistent limits.

## 7. Code Quality & Maintainability

🟡 **F-CQ-1 — Typecheck gate is red on `main`.** ✅ The `Typecheck` CI job fails on pre-existing errors (`superadmin`, `schools`, `AssessmentResult` shape, `BulkTools` selects). The "green gate" is currently not green. Fixes for 3 of 4 files are staged on `fix/typecheck-debt`; `superadmin` awaits a decision.

🟡 **F-CQ-2 — 132 `as any` casts** across 36 files actively hide DB-shape drift (worst: `BulkTools`, `CounselorStudentDetail` at 22 each). These are the silent siblings of the typecheck errors.

🟡 **F-CQ-3 — `ComprehensiveReportView` god component (~1,278 lines)** mixes data-fetch + scoring interpretation + rendering for all six instruments; high regression risk, untestable in pieces.

🟢 **F-CQ-4 — Thin automated tests.** Only a placeholder test plus the new Phase B scoring tests. Core scoring/auth/RLS paths are otherwise unverified by tests.

🟢 **F-CQ-5 — `useAuth` defensive sprawl.** Client-side auto-creation of role/profile rows and column-missing fallbacks (`preferred_language`) indicate the code copes with schema drift at runtime — tech debt that the schema/migration cleanup would let us remove.

🟡 **F-CQ-6 — Client-side role trust in `useAuth`.** ⓘ On a failed role fetch, the client sets the in-memory role from `user_metadata.role` (`useAuth.tsx` ~138–144), and auto-inserts a `user_roles` row from metadata (~55–71). Privileged escalation is **blocked server-side** by the live `enforce_role_assignment` trigger and RLS, so this is **not** a data-access bug — but it can render privileged UI to a non-privileged user (UI-level confusion) and is a defense-in-depth smell. *(Verified: dev role-override is correctly gated to `import.meta.env.DEV`.)*
